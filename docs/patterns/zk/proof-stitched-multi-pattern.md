# Proof-Stitched Multi-Pattern — Pattern 5.4

Status: DESIGN. Blocked on silverscript-lang exposing `OpZkPrecompile` and on KIP-21 sequencing-commitment lane stability.

## Summary

A composition primitive: one Groth16 proof verified once, shared across multiple covenant inputs in the same transaction via KIP-20's covenant-context opcodes (`OpCovInputCount`/`OpCovInputIdx`/`OpCovOutputIdx`). Lets a batch of covenants jointly attest to a single off-chain computation without each covenant paying the verification cost.

This is the **vProgs forward-compat target** for OpenSilver. vProgs aggregates many off-chain transactions into a single batch proof; Pattern 5.4 lets an L1 covenant cluster consume that batch proof in one verification rather than N. The pattern is what makes "L1-anchored vProgs settlement" affordable.

## Architectural sketch

```
Tx with N covenant inputs sharing covenant_id = C:
  - input[0]: leader  — runs OpZkPrecompile once
  - input[1..N-1]: delegates — check input[0] is the leader, no verification

  All inputs share state lineage via OpInputCovenantId(idx) == C.
  All outputs receive their share of the proof's released value, pinned
  via the proof's public inputs.
```

The leader/delegate split is the standard KIP-20 N:M shape from `DECL.md`. Pattern 5.4 specialises it: the leader runs the expensive `OpZkPrecompile` once; the delegates trust the leader and bind themselves to the same proof outcome via their shared covenant context.

## State layout (per-input)

```
verifyingKey     : byte[]    // shared across all inputs in the batch
batch_commitment : byte[32]  // root of the batch the proof attests to
n_public_inputs  : int
my_share         : int       // this input's share of the batch payout
my_recipient     : pubkey
```

## Intended `.sil` shape (compile-blocked)

Two entrypoints — `__leader_settle` and `__delegate_settle` — generated by the compiler from one policy function:

```sil
#[covenant(binding = cov, from = max_batch, to = max_batch, mode = verification)]
function settle(
    State[] prev_states,
    State[] new_states,  // empty for terminal payouts
    byte[] proof,
    byte[] public_inputs_concat,
    sig leader_sig
) {
    require(checkSig(leader_sig, prev_states[0].owner));  // leader gate

    // Verify the proof ONCE for the whole batch.
    require(OpZkPrecompile(
        0x20,
        prev_states[0].verifying_key,
        proof,
        prev_states[0].n_public_inputs,
        public_inputs_concat
    ));

    // Iterate inputs, pin each one's recipient + share against the proof's
    // public inputs.
    for (i, 0, prev_states.length, max_batch) {
        // public_inputs_concat[i * 64 + 0..32]  = expected recipient
        // public_inputs_concat[i * 64 + 32..64] = expected share
        require(prev_states[i].my_recipient == /* slice public_inputs */);
        require(prev_states[i].my_share == /* slice public_inputs */);
        // outputs[i] pays my_share to my_recipient (P2PK).
        require(tx.outputs[i].value == prev_states[i].my_share);
        require(tx.outputs[i].scriptPubKey ==
                byte[](new ScriptPubKeyP2PK(prev_states[i].my_recipient)));
    }
}
```

The delegate path generated by the compiler does NOT verify the proof again — it only asserts `OpCovInputIdx(cov_id, 0) == this.activeInputIndex` for the leader path, or `!= this.activeInputIndex` for delegates. The single Groth16 verification at the leader covers the entire batch.

## Composition with KIP-21

vProgs' on-chain bridge consumes the `accepted_id_merkle_root` (post-Toccata activation: `SeqCommit(B)`) as its anchor. A Phase 5.4 covenant cluster can use `OpChainblockSeqCommit(anchor_block_hash)` to pull a committed sequencing root and treat it as one of the proof's public inputs. That gives the batch proof an L1-attested anchor — no oracle committee needed for the freshness guarantee.

This is the path that makes vProgs settlement *trust-minimised on L1*: the proof attests to vProgs lane activity, the anchor is L1-canonical, and the covenant batch unfolds the per-recipient payouts.

## Cost amortisation

| Inputs in batch | Groth16 calls | Cost per recipient |
| --- | --- | --- |
| 1 | 1 | `Gram(140 * 1000)` |
| 4 | 1 | `Gram(35 * 1000)` |
| 16 | 1 | `Gram(8.75 * 1000)` |
| 32 | 1 | `Gram(4.4 * 1000)` |

Cost per recipient is `total_proof_cost / batch_size`. With 32 recipients in one batch, the proof cost amortises to roughly 3× ECDSA per recipient. This is what makes this pattern affordable at scale.

## Security considerations

- **Single point of failure on the leader.** If input[0] is somehow corrupted (e.g. leader's signing key is the only key in the cluster), the entire batch is compromised. Production deployments should derive the leader role from a stable rule (e.g. lowest input index, leader-elect signature) rather than a trust assumption.
- **Public inputs layout is the security boundary.** The contract slices `public_inputs_concat` to extract per-input expectations. A bug in the slicing (off-by-one, wrong stride) silently corrupts every recipient's payout. The SDK glue MUST own this layout authoritatively — pattern authors do not hand-roll slice indices.
- **Batch_size must be bounded by `max_batch` at compile time.** Loops in SilverScript require a compile-time upper bound. Production deployments must commit to a specific batch size at compile time; dynamic batch sizes need separate compiled artifacts per size.
- **Per-input value conservation.** The contract enforces that `sum(my_share[i]) <= input_value` so the batch can't pay out more than it received. This is a separate require() from the proof — the proof attests *correctness* of the share allocation, the contract attests *consistency* with the input value.

## When to use

- vProgs L2 → L1 batch settlement.
- Airdrops where one proof attests to the full recipient set.
- Cross-chain bridge settlement where the proof is a state-transition proof of the source chain.

## WHEN NOT TO USE THIS

- Do not use this for any production deployment until `kaspanet/rusty-kaspa#775` clears its `TODO(covpp-mainnet)` audit-pending marker AND KIP-21 advances from Draft.
- Do not use this for batch sizes below ~4 — the per-recipient cost savings only kick in at scale. Single-recipient settlements should use Pattern 5.1 directly.
- Do not use this when the leader role can be cheaply captured. The proof's correctness doesn't help if the leader can be bribed to submit a proof for a different batch.
- Do not use this until the silverscript-lang `OpZkPrecompile` builtin lands AND the SDK ships `sdk/zk/groth16.ts` with stack-order safety AND a batch-input-layout helper that owns the `public_inputs_concat` slicing rules.

## Audit status

Not implemented. Design-only. Most ambitious of the four Phase 5 patterns; should be the LAST one implemented in V1, after the other three have been audited and exercised on TN12.
